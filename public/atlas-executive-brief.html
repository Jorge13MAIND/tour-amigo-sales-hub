Security & Data Protection

Enterprise-Grade Security

ATLAS handles sensitive sales data. Every layer is designed with security as a first-class concern.

■ Row-Level Security (RLS)

All 18 Supabase tables have RLS enabled. The anon role can only SELECT (read) data. Only the service_role (Edge Functions) can INSERT/UPDATE/DELETE. No client-side writes to sensitive tables.

■ Credential Isolation

All API keys (HubSpot, Gmail OAuth, Slack, Supabase service key) are stored as Supabase Edge Function secrets. Never in code, never in git, never on the client. The atlas-proxy Edge Function is the sole gateway.

■ OAuth 2.0 for Google

Gmail and Calendar access uses OAuth 2.0 with refresh tokens. No password storage. Tokens auto-refresh via the atlas-proxy Edge Function. Scopes limited to gmail.modify and calendar.readonly.

■ No Direct Database Exposure

The Command Center connects to Supabase via the public anon key (read-only). All write operations go through authenticated Edge Functions. No direct SQL access from the browser.

■ Approval Gates

High-value actions (sending emails to enterprise clients, changing deal stages, pricing decisions) require Jorge's explicit approval through the Command Center. Agents cannot bypass this.

■ Full Audit Trail

Every agent action is logged in the agent_activity table with timestamp, action type, reasoning, and result. 316+ entries and counting. Complete traceability of every automated decision.

■ HubSpot as Source of Truth

Supabase is a cache layer, not the source of truth. All deal decisions are made against live HubSpot data. If Supabase goes down, agents fall back to direct HubSpot queries.

■ Graceful Degradation

If any service fails (Gmail, Calendar, Slack), agents skip that step and continue. They always deliver something useful, even if degraded. No cascading failures.

Risk Mitigation

What Could Go Wrong (and How We Prevent It)

Agent sends wrong email

Enterprise deals (Tier 1) always require approval before sending. Only Tier 2/3 auto-send is enabled, and only for playbook-matched follow-ups. All drafts are visible in Gmail drafts folder before sending.

API costs spike unexpectedly

Daily limit of 22 runs hardcoded in schedule. Each run uses Sonnet (not Opus) to control cost. Batch processing and caching reduce token usage by up to 50%. Monthly cap can be set via Anthropic dashboard.

Service outage (Supabase/HubSpot)

Each agent handles failures gracefully: skip the broken service, note it, continue with available data. Morning brief still delivers even if Gmail is down. No cascading failures.

Data becomes stale

Gmail cross-reference patch ensures agents check actual email activity before calling deals "stale." HubSpot syncs hourly via pg_cron. Deal monitor runs every 30 minutes.
